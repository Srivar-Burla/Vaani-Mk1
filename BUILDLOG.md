# Vaani Build Log

A chronological record of observations, decisions, and tradeoffs made while building Vaani. Newest entries at the bottom.

---

## Entry 0: Project origin, initial planning, and early tooling decisions (Steps 1 to 4)

**The starting point: a personal itch.** Vaani began with a recurring frustration: thinking of questions while walking or doing something hands on, phone in pocket, earphones on, and losing the curiosity by the time an app was opened and the question typed out. The base concept was a hands free voice interface to an LLM, triggered from TWS earphones. A live translation feature was layered on top to make it genuinely useful for an Indian context, where conversations around you can be in any of a dozen languages.

**Initial scope: three states, two command words.** The first full design was a state machine: sleeping (wake word listening), active (LLM conversation), and translating (live language translation, triggered and exited by voice commands). "Bhasha" to enter translate mode, "Bas" to exit, both detected via string matching on the transcript rather than separate wake word models, since training custom wake words was assessed as too time consuming. This three state design was the mental model carried into the actual build, even before any code was written.

**Tech stack was chosen before any code.** The deliberate framing was: wake word locally (cost, privacy), STT via Sarvam (saarika), translation via Sarvam (speech to text translate), LLM reasoning as a swappable component, TTS via Sarvam (bulbul). Sarvam's models are built specifically for Indian languages and accents, which made them the natural foundation for a product whose entire premise is multilingual Indian voice interaction. Wake word (originally planned via Picovoice/Porcupine) and the LLM layer were the two components not on Sarvam's stack, for cost and capability reasons respectively, both flagged early as the most likely points of friction.

**Picovoice's free tier had disappeared.** The first thing actually attempted from this plan was wake word, since it was assessed as the most technically uncertain piece. Picovoice, the default choice for Porcupine, turned out to have deprecated personal free tier accounts, leaving only a 7 day trial before a $6,000/year Foundation plan. Replaced with OpenWakeWord, a fully local, open source alternative with pre built models including "Hey Jarvis," requiring no API key and no recurring cost. First instance of a recurring theme across this build: a plan made on paper had to be revised against what was actually available, and the revision (OpenWakeWord) turned out to be a better long term fit for the local first framing than the original choice anyway.

**pyaudio could not be built on Windows.** Installing pyaudio for microphone capture failed with a Visual C++ Build Tools requirement, since it compiles a C extension from source. Rather than install several gigabytes of build tooling for one library, switched to sounddevice plus soundfile, which install via pre built wheels and cover capture and playback equally well.

**Windows Defender blocked Python 3.14's own compiled extensions.** The most disruptive issue of the early setup. Installing google-generativeai failed with "DLL load failed while importing cygrpc: An Application Control policy has blocked this file." The same error reappeared for sklearn, a dependency pulled in by openwakeword. Root cause: Python 3.14 is new enough that Windows Defender's reputation system has no trust history for these .pyd files and blocks them by default, even on a personal machine with no corporate policy involved. Fixed by adding Defender exclusions for both the Python install directory and the project folder, followed by a full machine restart, not just a VS Code restart, for the exclusions to take effect. Running VS Code as Administrator afterward reduces recurrence. This single issue cost more time than every other Step 1 to 4 issue combined, and is the kind of thing that would be invisible on Mac or Linux.

**Replaced google-generativeai with google-genai.** Separately from the Defender issue, moved away from google-generativeai, which depends on grpc, toward google-genai, a newer SDK using plain HTTPS. This sidesteps the grpc/cygrpc problem at the root rather than patching around it.

**Picked Gemini over Claude, DeepSeek, and Sarvam's own LLM, before writing a line of LLM code.** Claude was ruled out on cost grounds for a personal project, since Anthropic's API has no free tier. DeepSeek was ruled out over data handling concerns for an assistant that will eventually carry personal context. Sarvam's own LLM was set aside since it is not yet positioned for general reasoning tasks. Landed on Gemini 2.5 Flash via AI Studio, the only option with a genuinely indefinite free tier. Decided early to isolate this choice behind a single get_llm_response function, anticipating that it would need to change again later.

**sarvamai, not sarvam-ai.** Minor but easy to lose time on: the correct PyPI package name has no hyphen. `pip install sarvam-ai` returns no matching distribution.

**Sarvam STT needs a file, not an array.** The transcribe endpoint expects a WAV file, not a raw numpy array, so every recording is written to a temporary file via soundfile before the API call.

**Windows file handle locking broke the temp file cleanup.** After writing audio to a NamedTemporaryFile and sending it to Sarvam inside a with open(...) block, os.unlink() failed with "the process cannot access the file because it is being used by another process." NamedTemporaryFile holds its own handle separate from the with open() block, and Windows enforces this strictly where Linux and Mac do not. Fixed with an explicit tmp.close() immediately after writing, before the file is reopened for the API call.

**saarika:v2 was deprecated mid build.** The first STT call returned a clean BadRequestError naming the exact replacement model, saarika:v2.5. Five minute fix, but a reminder that Sarvam's API surface is actively moving and model names in any tutorial or doc snippet should be treated as provisional.

**language_code="unknown" was the right default, not a placeholder.** Passing "unknown" triggers Sarvam's automatic language detection across Indian languages. Given Vaani's premise, that either the user or someone near them could be speaking any Indian language, hardcoding a language code would have been actively wrong, not just less flexible. Confirmed the response object returns both .transcript and a populated .language_code (for example "te-IN") only when "unknown" was passed, meaning no separate detection call is needed later.

**First product decision surfaced by a technical test.** Testing STT across Hindi, Telugu, and Kannada returned accurate transcripts, but all in native script. This raised a real design question for Step 5 onward: whether to translate non English transcripts to English before sending to Gemini, and translate Gemini's response back to the user's detected language before TTS. Logged as the first decision in this project where a passing test still changed the plan.

**sd.wait() is not optional.** sd.rec() returns immediately since it runs asynchronously; without sd.wait(), the script proceeds against an incomplete audio array.

**A mistranscribed name, correctly attributed.** "Srivar" came back as "Sriward" in one test. Full sentences in the same session, across three languages, transcribed correctly, so this was logged as a likely microphone or proper noun issue rather than a flag on Sarvam STT's general reliability.

## Entry 1: Project setup and scoping (repo creation)

**Renamed the project from Jarvis Mk1 to Vaani Mk1** before creating the GitHub repo. Renaming before the first commit meant no messy history rewrites later.

**Scoped wake word detection out of Mk1.** Initial plan assumed always-on wake word detection ("Hey Jarvis") via OpenWakeWord. Realised during planning that always-on mic scanning is heavily battery-intensive, and companies like Apple and Google solve this through deep hardware integration that commodity TWS earphones do not have. This is likely why OpenAI and Anthropic have not shipped voice invocation either. Decision: Mk1 uses a TWS gesture (button press) for invocation. Wake word moves to Mk2, conditional on a viable low-power approach.

**Scoped the Bhasha translate mode out of Mk1.** The live translation feature (for example, understanding an auto driver speaking Tamil) has an unresolved hardware problem: earphone mics are tuned to pick up the wearer's voice, not the opposite party's. Shipping a flaky version would be worse than not shipping it. Deferred to Mk2 pending real-environment testing.

**Separated README and PRODUCT.md.** README covers what the project is and how to run it. PRODUCT.md covers the problem, the reasoning behind the stack, and the build plan.

## Entry 2: Gemini integration (Step 5)

**Chose Gemini 2.5 Flash as the Mk1 reasoning layer** primarily because it is the only major LLM provider offering free API access, which makes development and testing practical without upfront cost. The LLM is deliberately isolated behind one function (get_llm_response) so swapping providers later touches one place.

**Debated and parked internet grounding.** Without live web access, hallucination risk on factual queries is real, and grounded responses can cite sources. But grounding adds a search step to an already 4-call-per-turn pipeline (latency), and each provider implements it differently (Gemini grounding is Gemini-specific), which would break the clean swappability design. Decision: ship Mk1 without grounding, feel the latency and the "I don't know" responses in practice, then treat grounding as P0 for Mk2. Longer term plan: one integration function per major provider, plus one for locally hosted open source models.

**Identified persistent memory as a future feature.** Chat sessions hold context within a conversation but reset between sessions. A real assistant needs to remember tasks, people, and personal context across days. Added to Mk2.

**First real debugging war story.** Passing the STT result object directly into Gemini's send_message raised a type error: the method expected a string but received an object. Initial hypothesis (client initialisation format) was wrong. Tracing the data flow showed the Sarvam STT response is an object with .transcript and .language_code fields, and only the transcript string should cross the API boundary. Lesson: every API boundary is a contract, and type errors usually point at exactly which contract was violated.

## Entry 3: TTS and the misread error (Step 6)

**Vaani spoke for the first time.** Sarvam TTS (bulbul:v2, speaker anushka) returns audio as Base64 inside JSON, since JSON cannot carry raw binary. Decode, write to a temp WAV, play through sounddevice. Latency felt acceptable.

**Misdiagnosed a 503 as a code bug.** First Telugu test crashed with a long traceback. Initial conclusion: "Telugu broke it." The actual final line of the traceback was a 503 UNAVAILABLE from Google's servers (model overloaded). The request never reached Gemini at all, and the same code worked minutes later. Lesson: read the error before forming the hypothesis, and distinguish server-side failures from client-side bugs.

**Discovered a product requirement from the crash.** A voice assistant cannot show a stack trace. Any API failure should be caught and spoken: "sorry, I'm having trouble right now." Error handling is a core requirement for a screenless product, not a polish item. To be built with the conversation loop.

**A surprise that challenged the architecture.** With the TTS language hardcoded to en-IN, a Telugu run worked anyway: Gemini responded in Telugu script and bulbul:v2 spoke it, with a slightly off accent. This raised the question of whether the translation legs are needed at all (skipping them would cut 2 API calls per turn). Decided to keep the translation pipeline for two reasons: Gemini's response language is inconsistent for non-English input (sometimes native script, sometimes Latin script, sometimes English), and LLM reasoning quality is strongest in English. Logged a future experiment: an isolated TTS test harness fed with real Gemini outputs of varying scripts, run only if Mk1 latency proves too high.

## Entry 4: The full multilingual loop (Step 7)

**Wired both translation legs.** The STT-detected language_code drives everything: whether to translate inbound, whether to translate outbound, and the TTS target language. English queries skip both translation calls entirely, keeping the common case fast. Tested end to end in English, Hindi, and Telugu.

**System prompt iteration after real usage.** The first prompt draft said "be decisive, do not front-load uncertainty," yet Vaani still opened answers with disclaimers ("the best is subjective...") and once demanded trip details before estimating a drive time. Diagnosis: caveat behaviour is different from uncertainty hedging, and LLM training rewards thoroughness that voice cannot afford, because the listener must sit through every word. Fix: explicit instructions to never open with disclaimers, to answer first and caveat after in one short phrase only if it matters, and to make stated assumptions instead of asking follow-up questions. Before/after difference was immediate and large. Lesson: system prompts are never done after one draft; real usage exposes the gaps.

## Entry 5: Silence detection design (Step 8)

**Replacing the fixed 5-second recording with dynamic silence detection.** The recording should end when the user stops talking, not at an arbitrary timer. Core approach: record in small chunks, measure each chunk's loudness, and end the turn after roughly 2 seconds of continuous silence.

**Why 2 seconds of silence, not instant.** Natural speech has gaps: pauses between words, and mid-query pauses while the user thinks through how to frame a question out loud. Unlike chat, where the user composes and edits before hitting send, voice has no explicit "done" signal, so the system must infer it. The silence window has to be long enough not to cut off a thinking pause, short enough not to feel sluggish. This is a product tuning decision, not just a technical constant, and likely belongs in user-configurable settings eventually (a fast talker and a deliberate thinker want different values).

**Measuring loudness: mean of absolute amplitudes.** An audio chunk is a numpy array of amplitude values that swing positive and negative. Averaging them directly cancels out to near zero even for loud chunks, since the wave is roughly symmetric. Taking the absolute value first (or RMS) fixes this. Mean-of-absolutes is sufficient for distinguishing speech from silence.

**Fixed threshold for Mk1, adaptive threshold deferred to Mk2.** A fixed loudness threshold calibrated in a quiet room will misbehave in noisy environments: ambient noise on a street could read as speech. The correct long-term design is an adaptive threshold that samples the ambient noise floor and sets the speech cutoff relative to it. This works because the mic sits close to the user's mouth, so their voice arrives significantly louder than background noise in almost any environment, leaving a clear gap to detect against. Building the adaptive version now would mean optimising a component before the full conversation loop exists, so Mk1 uses a fixed threshold derived empirically (see the measurement script), with the adaptive approach logged for Mk2.

**Empirical threshold over guesswork.** Rather than guess what loudness value separates speech from silence, wrote a standalone measurement script that records and prints the volume of each chunk, making the speech-versus-silence gap visible in real numbers. The Mk1 threshold is read off that data.

## Entry 6: Silence detection, multi-segment playback, and TTS pronunciation (Step 8)

**Dynamic silence detection replaces the fixed recording window.** Recording now ends after roughly 2 seconds of continuous silence rather than a fixed timer, using a record-in-chunks loop with a has_spoken gate (so leading silence before the user starts is ignored) and a silence counter that resets on any loud chunk (so mid-sentence thinking pauses do not end the turn early). Threshold and silence window were derived empirically from a standalone measurement script (volume_measurement.py) rather than guessed. Speech sits around 0.01 to 0.03, ambient and pauses around 0.0005, so a threshold of 0.005 sits cleanly in the gap.

**Loudness is measured as the mean of absolute amplitudes.** An audio chunk is a numpy array of values that swing positive and negative, so averaging them directly cancels to near zero even for loud chunks. Taking the absolute value first fixes this. A fixed threshold is used for Mk1, calibrated in a quiet environment. An adaptive threshold that samples the ambient noise floor and sets the speech cutoff relative to it is the more robust design (it would hold up in noisy environments, and works because the mic sits close to the user's mouth so their voice is always significantly louder than background), but it was deferred to Mk2 to avoid optimising a component before the full loop exists.

**Long responses were being cut off in speech.** Sarvam TTS returns long text as a list of audio segments, and the original code only decoded and played the first segment (audios[0]), so anything past the first chunk was silently dropped while the terminal still showed the full text. First fix attempt, concatenating the decoded bytes into one blob, failed with a "No RIFF chunk marker" error: each segment is a complete standalone WAV file with its own header, so raw byte concatenation produces a malformed file. Correct fix: loop over the segments and play each complete WAV in sequence, with sd.wait() ensuring each finishes before the next begins. Lesson: a list of WAV files is not the same as one long WAV.

**Alphanumeric codes needed explicit prompt handling.** "NH44" was spoken as a jumble. The system prompt already said to avoid acronyms, but did not cover alphanumeric identifiers. Added an instruction to spell out codes the way a person says them aloud ("National Highway forty four", not "NH44"). Worked immediately. Noted the limit of this approach: prompt-patching every code pattern does not scale, so a deterministic text normalization pass between the LLM and TTS is the more robust Mk2 solution.

**English TTS pronunciation is a model-level limitation in Bulbul, investigated and deferred.** Bulbul handles Indic scripts well but its English pronunciation (rendered from Latin script) is uneven: "Chennai" comes out with a near-silent C, "I" sounds like "ye", "6 AM" becomes "6 way", and even "Krishnagiri", a Telugu-rooted place name, is mispronounced because it arrives romanized in Latin script rather than in Telugu script. Root cause is consistent across all of these: Bulbul's weak path is English from Latin characters, which is precisely the path most users will hit, since English is expected to be the dominant usage language. This cannot be compromised for Mk1's actual user base, so it was treated as a real problem rather than a defer-by-default.

Investigation ruled out the cheap fixes. Tested every Bulbul v2 speaker, multiple female and male voices, and the same failures held across all of them, confirming it is model-level and not a voice-selection issue. Also confirmed the language parameter is not the cause: removing target_language_code (defaulting to auto-detect) did not change the pronunciation, since Latin-script English resolves to English regardless. The real fix is routing the English path to a second TTS provider with strong Indian-English voices (Google Cloud TTS or Azure) while keeping Sarvam for Indic-script output, behind a single provider-agnostic speak() function so main.py does not need to know which engine runs.

This was then deferred deliberately, not for lack of a solution but because every candidate provider requires cloud billing setup as a hard gate. Google Cloud requires an active billing account attached to a project even to use the always-free tier, and UPI in India is a prepaid, organization-oriented method rather than a zero-balance option, so there is no way to attach it without committing funds or a card. That is pure setup friction with no product learning, and it was blocking progress on the core conversation loop. Decision: finish the conversation loop and the rest of Mk1 first, then return to the English TTS provider integration at the end of Mk1 or in Mk2, with the architecture already designed and the diagnostic work already done.

## Entry 7: The conversation loop (Step 9)

**Vaani became conversational.** The single-turn script is now wrapped in a while True loop, turning a one-shot demo into a back-and-forth assistant that listens again after every response. The setup (clients, system prompt, function definitions, and crucially the chat_session) lives once outside the loop; only the per-turn work (record, STT, translate, reason, translate back, speak) repeats inside it. Keeping chat_session outside the loop is what gives Vaani memory across turns: it accumulates the full conversation transcript, so follow-up questions that depend on earlier context work correctly. Tested with multi-turn sessions where later questions relied on the running conversation.

**Refactored the speak logic into a reusable function.** While converting to a loop, the translate-back-and-speak block needed to run in two places: for normal responses, and for the goodbye message on exit. Rather than duplicate it, extracted it into a vaani_output(output, user_lang_code) function that handles the outbound translation (skipping it for English) and the multi-segment TTS playback. The exit branch now reuses this same function with a fixed goodbye string, so the sign-off is spoken in the user's own language with no duplicated code.

**Spoken exit command, with a fixed goodbye.** The conversation ends when the user says an exit phrase ("goodbye", "bye", "stop", "that's all for now"), detected by lowercasing the translated English text and checking with any() against a list of phrases. The exit check sits after inbound translation but before the Gemini call, so saying goodbye does not waste an LLM request. The goodbye line itself is a fixed string, not generated by Gemini (no reason to spend an API call or latency producing a predictable sign-off), and it is routed through vaani_output so it is spoken in the user's language.

**Known limitation: substring matching on exit phrases can cause false exits.** Exit detection uses substring matching, so a phrase like "stop" would also match inside a legitimate question such as "should I stop at the next exit", ending the conversation unintentionally. Acceptable for Mk1 with deliberately chosen exit words, but a more robust version would match whole words, or require the exit phrase to be the entire short utterance rather than a fragment inside a longer one. Logged for a future refinement.

**Performance note: optimise the bottleneck, not the cheap code.** While building the exit check, considered the cost of the any() substring scan. Concluded it is negligible (a handful of phrases against a short sentence, microseconds) next to the real cost of each turn, which is the network round-trips for STT, translation, Gemini, and TTS. General principle carried forward: in this pipeline the API calls dominate latency, so optimisation effort belongs there, not on local string or array operations.

## Entry 8: Exit-detection robustness and a naming consideration

**Exit detection: from substring matching to whole-utterance, and the limit beyond.** The first conversation loop detected exit phrases with substring matching (phrase in text). This has a false-positive risk: "stop" matches inside "should I stop at the next exit", which would end the conversation mid-thought. Tightened the Mk1 version to match the whole cleaned utterance (lowercased, trailing punctuation stripped) against the exit list, optionally allowing a short utterance that contains an exit word, so deliberate exits work but exit words buried in real questions do not.

Then hit the real ceiling of the string-matching approach. Natural conversation endings like "thanks, that's all for now", "great, I'll let you know if I need anything else", or "no need, that's all" are how people actually wind down, and they defeat every string-based method: some are long multi-word sentences, and some ("I'll let you know if I need anything else") contain no exit keyword at all. Ending a conversation naturally is a semantic intent signal, not a lexical one, so it cannot be enumerated as phrases. Decision: Mk1 keeps reliable explicit-exit-word matching (the user is told the exit words at startup, and it never fires by accident); detecting natural wind-downs is deferred to Mk2 as LLM-based intent detection, since the model is the right tool for judging intent and that is a meaningfully larger change to how the exit decision is made.

**Naming consideration logged for the future.** "Vaani" (वाणी, "speech/voice" in Sanskrit) is a meaningful and on-theme name, but it is heavily used in this exact space: a funded Bengaluru voice-AI startup is named Vaani AI, and both Meesho and UTI Mutual Fund launched products called Vaani in early 2026, plus a few open-source projects share the name. Not a problem for a personal portfolio project (the word is generic and nothing is being sold), but it hurts discoverability if the project is ever published or talked about more widely, since it competes for attention against larger namesakes. Keeping Vaani through Mk1 for its meaning and continuity. If a rename is ever warranted, candidate Sanskrit-rooted alternatives worth checking for usage first: Dhwani (sound/resonance), Bodha (understanding/comprehension, which maps to the project's actual differentiator of understanding rather than just transcribing), Shruti (that which is heard), and Sakhi (companion). A usage check is essential before adopting any, since meaningful Sanskrit words are a common naming choice for Indian AI products and collisions are likely.

## Entry 9: Error handling across the pipeline (Step 10)

**Every API call in the loop now fails into speech, not a crash.** A screenless assistant that dies with a traceback is invisible to its user, who is left talking to silence with no idea what happened. The conversation loop is now resilient: each risky call (STT, inbound translation, the Gemini request, and the response and goodbye output, which themselves bundle outbound translation and TTS) is wrapped in its own try/except, and every failure path ends in continue, which abandons the broken turn and returns to listening. One bad turn no longer ends the session.

**Tailored messages for known failures, a generic apology for everything else.** Rather than one catch-all line, failures are inspected and matched to a spoken response. The main known case is the Gemini rate limit, split further into two genuinely different user situations: the daily free-tier pool being exhausted ("I've used up all my free questions for today, I'll be ready again tomorrow") versus a short-term rolling limit ("give me a minute, then try again"). The split keys off the metric string in the error text, since GenerateRequestsPerDay survives lowercasing as perday, so it reads off something real rather than guessing. A connection-error branch covers network drops, and a generic apology catches everything unknown, including the 503 server-overload that surfaced during testing (a transient Google-side high-demand error, distinct from a quota 429).

**Layered degradation handles the case where Vaani cannot even speak the apology.** The apology is voiced through the same output path that may itself be failing, so the helper that speaks errors wraps its own output call in a nested try/except that falls back to a printed message and does not call the error speaker again. Speak the apology if possible, print it if not, never recurse, never crash. The honest limitation, logged rather than hidden: because the apology is voiced through Sarvam, a total Sarvam outage means it cannot be spoken at all and only prints, which is invisible on a screenless device. The deferred local audio cues (non-speech tones generated with numpy, needing no API) are the real fix for that gap, since they work even when every network call is down.

**Scoped the hands-free feedback layer out into its own batch.** Audio cues (listening, thinking, shutdown, error tones), the TWS gesture invocation, and a first-run setup UI were grouped together as the hands-free experience layer and deferred as one unit, to keep the urgent and bounded error-handling work from waiting on open-ended design.

**Temp file cleanup made crash-proof with finally.** The STT block writes the recording to a temporary WAV before sending it. Its cleanup now sits in a finally block, so the file is deleted whether the call succeeds or throws, preventing a leaked temp file on every failed turn, since continue would otherwise skip past the old cleanup line.

## Entry 10: Conditional web grounding, pulled forward from Mk2 (Step 11)

**Gemini can now search the live web, and decides for itself when to.** Until now Vaani had no internet access, so the system prompt instructed it to say plainly that it has no live information for questions about current events, prices, or anything recent. That was the honest stopgap. Grounding lifts the limitation for factual and recent queries: Gemini's native search tool is attached to the chat session, and the model itself decides, per turn, whether a question needs a live lookup, grounding its answer in real results when it does.

**Conditional grounding is what made this an Mk1 build rather than a Mk2 deferral.** The original reason for deferring grounding was latency, since it adds a search round-trip to an already multi-call pipeline. The realisation that changed the call: the model only searches when it judges the question needs it, so the common case, a question answerable from the model's own knowledge, pays nothing, and only genuinely live questions take the extra round-trip. The cost lands where it earns its keep instead of on every turn, which resolved the tension that had pushed grounding to Mk2.

**It comes at the cost of an explicit exception to the swappable-LLM principle.** The grounding runs through Gemini's own search tool, which is Gemini-specific, not a provider-agnostic mechanism. So the clean "swap the LLM by changing one function" design now has a documented exception: moving to another provider would mean re-implementing grounding against that provider's own grounding or tool-calling system. Accepted knowingly, because grounding is valuable enough to the product to justify the coupling, and the coupling is contained and written down rather than silent.

**Verified live.** Tested with a question about the Royal Enfield Himalayan 450, a recent motorcycle, routed to Chennai, which pairs a recent-product fact with a routing question. The model chose to search, grounded its answer, and returned correct live information, confirming both that the tool is wired correctly and that the model's self-decision to search fires on the right kind of query.

## Entry 11: Swapped the reasoning model to Gemini 3.1 Flash-Lite (Step 12)

**Changed the LLM from Gemini 2.5 Flash to Gemini 3.1 Flash-Lite, and it cost one line.** The only edit was the model string in the chat_session creation, because the LLM was isolated behind get_llm_response and a single model constant from the very first Gemini work in Entry 2. This is the swappability design paying off in practice rather than in theory.

**The driver was free-tier quota, confirmed against the live dashboard, not assumed.** The daily request cap had shaped sequencing decisions across the whole build, since 2.5 Flash on this project allowed only 20 requests per day, which is roughly four demo takes. The AI Studio rate-limit view showed 3.1 Flash-Lite on the same project at 500 requests per day and 15 requests per minute, against 20 per day and 5 per minute for 2.5 Flash. That is a twenty-five-fold increase in daily headroom, which effectively removes quota as a constraint on rehearsal and testing.

**Quality was a lateral-or-better move, not a downgrade, because the baseline was 2.5 Flash.** Google's own positioning is that 3.1 Flash-Lite matches 2.5 Flash quality while being faster and cheaper. The earlier worry about Flash-Lite being weaker applied only against the larger 3 Flash, not against the model actually in use. Flash-Lite is additionally well-suited to the structured extraction the transaction feature needs, since extraction and classification are its stated strengths.

**Two things to verify on any 3.x swap, logged so the next model change is faster.** Gemini 3 models think by default and expose a thinking_level parameter, so if a turn feels slow, setting it to low or minimal trades reasoning depth for voice-appropriate speed. And 3.x no longer recommends setting temperature, top_p, or top_k. Neither was being set here, so the swap did not break, but both are worth knowing before tuning. Grounding was re-confirmed on the new model rather than assumed to carry over.

## Entry 12: Transaction recording, Vaani's first real action (Step 13)

**Vaani can now take an action, not just answer.** Until now every turn ended in speech. A new flow, triggered when the user says a phrase like "record a transaction", turns a spoken description into a structured record and writes it to the personal finance tracker API over HTTP. This is the shift from voice interface to voice agent, and it ties the two separate projects together: Vaani is now the voice front end and the finance tracker is the backend it acts on. It is also the first point in the build where the role flips from owning the receiving end of an API, which is the day job, to being the caller constructing and sending the request.

**Chose hybrid extraction over pure slot-filling.** Rather than interrogate the user field by field, the flow first asks for a single natural description and runs a one-shot Gemini extraction that pulls every field it can find at once, returning JSON. Only then does it fall back to asking a spoken question for each required field that came back empty. So "four hundred on lunch at Empire by UPI" fills four fields in one turn, and the user is asked nothing further, while a vaguer description still gets completed through targeted questions. This is better to use and more convincing to demonstrate than a fixed four-question script.

**Required versus optional fields was a deliberate product cut.** Name, amount, category, and payment medium are required and will be asked for if missing, because those four are the minimum for a record worth querying later. Payment source and notes are optional: captured if the user volunteers them in the description, never asked for, because forcing them through spoken question-and-answer would bloat the flow for marginal value. The voice flow never collects datetime or id, consistent with the finance tracker's existing rule that those are server-captured and auto-generated.

**Extraction runs off the conversational session on purpose.** The extraction calls use a separate generate_content call rather than the chat_session, so the back-and-forth of pulling out fields does not get written into Vaani's conversational memory. The chat_session accumulates every send_message, so routing extraction through it would have polluted later conversation with bookkeeping turns. Same isolation instinct as keeping the LLM behind one function, applied to memory.

**Read-back confirmation is the safety gate for voice-to-money.** Before anything is written, Vaani reads the assembled transaction back ("I've got four hundred rupees at Empire, under Food, paid by UPI, shall I save this?") and waits for a yes. People forgive an assistant that mishears a question and do not forgive one that silently logs the wrong amount to a real ledger, so for a feature that writes financial records the confirmation is not polish, it is what makes the feature safe to ship. A no cancels cleanly without writing anything.

**JSON parsing is hardened against the model wrapping its output.** Gemini sometimes returns the JSON inside markdown fences despite being told not to, so the parser slices from the first opening brace to the last closing brace before parsing, rather than trusting the response to be clean JSON. A small but necessary defense, since the extraction is only useful if it parses every time.

**Errors in this flow stay in the session, by design.** Unlike the main conversational loop, a failure anywhere in the transaction flow is caught locally, spoken as a brief inline message, and control returns to listening rather than ending the conversation. A failed expense log is a small, recoverable event and should not cost the user their whole session. The POST itself uses a ten-second timeout and raises on any HTTP error, so a finance-API problem surfaces as that inline message rather than a silent non-write.

## Entry 13: Session logging with a stdout tee (Step 14)

**Every session now writes a full log with no change to the rest of the code.** A small Tee class wraps stdout and stderr so that every print already in the program, the user transcript, the detected language, the English intermediate, Vaani's reply, the extracted transaction fields, and every traceback, is written to both the terminal and a timestamped file under a logs folder. Because the existing print statements were already acting as a trace of each turn, teeing the streams turned them into a persistent record for free, with no new logging calls scattered through the logic.

**Both streams are teed so errors are captured too.** stderr is teed alongside stdout specifically so that tracebacks, including those printed inside speak_error, land in the same file as the normal flow, which is what makes a log useful when something fails mid-session. The file is opened once per run with a timestamped name and closed cleanly on exit via atexit. This is the observability layer that makes debugging the multi-API pipeline after the fact possible, and it doubles as a record of demo runs.

## Entry 14: Revised error recovery from continue to exit, based on observed failures (Step 15)

**Changed the conversational loop's error behavior from "apologise and keep listening" to "sign off and exit," which revises Entry 8.** Entry 8 built error handling so that a failed turn ran continue and returned to listening, on the principle that one bad turn should not end the session. Real usage changed that conclusion. The failures actually encountered were almost entirely Gemini daily-quota exhaustion and Gemini 503 high-demand, and neither resolves in seconds: the daily cap resets only at midnight Pacific, and the 503 payload prescribes a minimum wait of around sixty seconds before retrying. Returning to listening after either would simply re-hit the same wall within seconds and trap the user in a loop of identical apologies. So for these cooldown-bound failures, exiting with a spoken sign-off is the more honest behavior than pretending the next turn might work. The spoken messages in speak_error were rewritten accordingly, from mid-conversation apologies into graceful sign-offs that tell the user when to come back.

**The transaction flow deliberately keeps the opposite behavior.** A failure inside transaction recording is caught locally, spoken as a brief inline message, and returns to listening without ending the session. This is intentional and not an inconsistency: a failed expense log is a small, recoverable event unrelated to the assistant's core availability, whereas a failed Gemini call in the main loop means Vaani cannot reason at all for at least a minute. The two flows differ because the failures differ in kind.

**Accepted tradeoff, logged honestly.** The exit path is uniform: any unhandled error in the conversational loop exits, not only the two Gemini cases that motivated the change. This is a simplification. A genuinely transient non-Gemini hiccup, such as a momentary Sarvam or network blip, would also end the session even though a retry on the next turn might have recovered. This was accepted over the added complexity of distinguishing cooldown-bound failures from instantly-retryable ones. Noted as a possible refinement: a transient 503 could speak-and-continue rather than hard-exit, which would matter most in a live demo where a full restart is the worst outcome. Not built, deliberately, since the observed failures are overwhelmingly the cooldown-bound kind.

## Entry 15: Added a desktop interface without replacing the developer terminal flow (Step 16)

**Added a minimal Tkinter interface as a second entry point while keeping main.py available for development and troubleshooting.** The interface shows whether Vaani is Idle, Listening, Thinking, or Speaking, provides a running conversation transcript, and exposes a Start Listening button. Non-English turns show both the native script and the English text passed to or returned by Gemini, which makes the multilingual translation boundary visible without filling the interface with internal debug output. Each new conversation clears the previous transcript, and the Start button becomes available again when the conversation ends.

**Moved the blocking conversation loop onto a background thread so the interface remains responsive.** Tkinter must own the main thread, while microphone capture, API calls, translation, and speech playback can block for several seconds. The conversation pipeline therefore runs on a worker thread and sends state and transcript messages through a queue that Tkinter polls from its own thread. The button and later gesture invocation share one trigger function, and a session-active guard prevents overlapping conversations. Running main.py directly still gives the developer the original terminal and log output, while running ui.py gives the user the desktop experience with output written only to the session log.

**Kept the first interface deliberately small.** Tkinter was chosen because it is part of Python's standard library and avoids introducing a GUI framework dependency. The interface is not intended to replace the voice-first experience with a dashboard. Its role is to expose the assistant's current state, provide a readable transcript, and preserve a reliable manual trigger when an earbud gesture is unavailable.

## Entry 16: Implemented TWS invocation through a WinRT media session (Step 17)

**TWS Play and Pause gestures now invoke the same listening flow as the desktop button.** The working implementation creates a WinRT MediaPlayer and obtains its System Media Transport Controls session, then listens for Play and Pause events sent by Windows. Detected gestures are placed on the existing Tkinter queue, so the UI thread safely calls the shared trigger function. If a conversation is already active, the existing session guard ignores additional taps rather than attempting to interrupt microphone capture, an API request, or speech playback.

**Several lower-level Windows input approaches were rejected after direct testing.** A pynput keyboard hook received ordinary keyboard presses but no Bluetooth earbud events. WM_APPCOMMAND and RegisterShellHookWindow received only window redraw side effects, not the actual media command. Raw Input also received nothing. The reason is that these earbuds send Bluetooth AVRCP commands through Windows System Media Transport Controls rather than through the keyboard, shell, or raw HID input paths. A direct SMTC call using get_for_current_view also failed because an unpackaged Python application has no UWP view. Creating a MediaPlayer solved that constraint by giving the application a valid media session inside normal Win32 Python.

**Accepted the media-session tradeoff because hands-free invocation is central to Mk1.** While Vaani is running, Windows may route the earbud Play or Pause gesture to Vaani instead of Spotify, YouTube Music, or another media application. The alternative was to abandon the TWS interaction that defines the intended hands-free demonstration. The desktop Start Listening button remains available as a fallback, and the WinRT listener runs on a daemon thread so it does not block the interface or the existing STT, translation, Gemini, translation, and TTS pipeline.

## Entry 17: Reverted the conversational model to 2.5 Flash because grounding is incompatible with 3.x on the free tier (Step 18)

**Moved the chat model back from Gemini 3.1 Flash-Lite to Gemini 2.5 Flash, which partially supersedes Entry 11.** Every conversational turn began failing with a 429 RESOURCE_EXHAUSTED, and it was reproducible immediately on a fresh API key, which ruled out ordinary quota burn from the start. The cause was not input size, not token count, and not the daily cap. It was that the search-grounding tool from Entry 10 returns an instant 429 on the 3.x model family on this free tier, while the same tool works on the 2.5 family. Entry 11 had swapped the model to 3.1 Flash-Lite for its much larger daily quota, and that swap is what silently broke grounding. Reverting the chat model restores a working assistant. Only the conversational chat session is reverted; the transaction-extraction calls stay on 3.1 Flash-Lite, since they never attach the search tool and so were never affected.

**The diagnosis came from isolating the one variable that mattered: the tool, not the model.** A throwaway script ran the same minimal prompt four ways. On 3.1 Flash-Lite without tools it succeeded, and on 3.1 Flash-Lite with the search tool it returned 429. Across families, 2.5 Flash and 2.5 Flash-Lite both grounded successfully while 3.1 Flash-Lite and the flash-latest alias both 429'd. That pattern is conclusive: the rejection tracks the tool plus the 3.x family, not the model alone and not the prompt. The reason a question that needed no live lookup still failed is that the search tool is attached to the chat session, so every request declares it regardless of whether the model would actually choose to search, and the rejection happens before any such decision.

**This corrects a claim in Entry 11.** Entry 11 stated that grounding was re-confirmed on the new model rather than assumed to carry over. Whatever was observed at the time, grounding on 3.1 Flash-Lite does not hold on this free tier now, so that re-confirmation did not catch the regression. The honest lesson is that grounding compatibility is a property of the specific model-and-tier combination and has to be tested against the exact failing path, with the tool attached, not inferred from a model working in isolation.

**The accepted tradeoff is giving back the daily headroom Entry 11 bought.** The whole point of Entry 11 was that 2.5 Flash allowed only about twenty requests per day against 3.1 Flash-Lite's five hundred, and that cap had shaped testing and demo sequencing across the build. Reverting puts the conversational path back under the tighter cap. That is accepted because a grounded assistant that answers at a low daily limit is strictly more useful than a higher limit that fails on every grounded turn, and grounding is a shipped Mk1 capability rather than a nice-to-have. The extraction calls deliberately stay on 3.1 Flash-Lite so their volume does not consume the conversational model's scarce daily quota. The two alternatives considered and rejected were enabling billing to unlock 3.x grounding, which adds real per-query cost to a deliberately free personal project, and dropping grounding to keep the higher quota, which would remove a capability the product was specifically built around in Entry 10.

**A separate observation worth fixing later: the spoken error message misdescribes this failure.** The message told the user it had hit a token limit and processed too much text, when the actual error is a grounding-quota rejection unrelated to text volume. That wording sent debugging in the wrong direction, toward input size and quota top-ups, rather than toward the tool. Correcting what Vaani says on this class of error is a user-facing change and is left for a separate decision rather than folded into this revert.

## Entry 18: Designed a selective-grounding router, then deferred it to Mk2 to protect the swappable-LLM design (Step 18)

**Reverting to 2.5 Flash in Entry 17 fixed grounding but reintroduced a hard ceiling, and the free tier offers no single model that escapes it.** On this tier every grounding-capable model, both 2.5 Flash and 2.5 Flash-Lite, is capped at about twenty requests per day, while the only models with a high daily cap, the 3.x family at five hundred, cannot ground at all. Grounding and daily volume are therefore mutually exclusive within any one model, so the twenty-per-day cap is not a tuning choice but a structural property of grounding on the free tier. An initial hope that 2.5 Flash-Lite might pair grounding with a higher cap was checked against the live rate-limit dashboard and proved false: it shares the same twenty-per-day ceiling as full 2.5 Flash.

**To get both, we designed a two-model router and worked it out in full before deciding.** The shape was 3.1 Flash-Lite as the primary brain, answering every ordinary turn from its five-hundred-per-day budget and, in the same call, judging whether a question needs live data by emitting a [SEARCH] sentinel instead of an answer. Only flagged queries would reroute to 2.5 Flash with grounding, spending one of the scarce twenty, while Vaani spoke a short announcement to the user to cover the extra lookup latency. That would have made casual conversation effectively unlimited and reserved the twenty daily grounded calls for the queries that genuinely need them, which is exactly the selectivity the cap demands.

**It was deferred to Mk2 because it pays for a quota optimization with the one principle the whole build protects.** The router hard-codes a two-model, Gemini-specific routing path, the sentinel convention plus Gemini's native search tool, straight into get_llm_response, the single function the architecture keeps provider-agnostic so the LLM stays swappable. That deepens the grounding coupling already flagged as an exception in Entry 10 rather than containing it, and a later move to Claude or a local model would have to re-implement not just grounding but the routing logic around it. It also forces abandoning the stateful chat session for manually managed history, so the [SEARCH] sentinel does not pollute the model's record of what it actually said, which is a real rewrite of the LLM layer rather than an isolated addition. Set against a benefit that is only a free-tier quota workaround, the trade was judged wrong for Mk1.

**The accepted position is that twenty grounded turns a day is livable, and the router belongs where the swappable-LLM work already lives.** Mk2 builds dedicated per-provider integrations anyway, so a provider-agnostic grounding-and-routing abstraction can absorb the router cleanly there instead of bolting a Gemini-specific path onto Mk1. A secondary reason it was not a clear win even on its own terms: the primary model's self-judgment of when it needs to search is inherently imperfect, so even built it would sometimes answer a stale question from memory rather than flagging it, meaning the feature would ship with a known accuracy gap on top of its architectural cost.

## Entry 19: Enabled billing, switched the conversational model back to 3.1 Flash-Lite, and obviated the Mk2 router (Step 18)

**Enabled billing on the Gemini project, which dissolves the constraint behind Entries 17 and 18.** Loaded a thousand rupees of credit, moving the project off the free tier. The 429 on grounded calls to the 3.x family was a free-tier restriction rather than a model limitation, so before changing any code we re-ran the exact combination that had failed, 3.1 Flash-Lite with the Google Search tool attached, and it now returns a correctly grounded answer. The verify-the-failing-path discipline from Entry 17 carried over rather than assuming billing fixed it.

**Switched the conversational model back from 2.5 Flash to 3.1 Flash-Lite, superseding the Entry 17 revert.** Entry 17 moved to 2.5 Flash only because 3.1 Flash-Lite could not ground on the free tier. With that blocker gone, 3.1 Flash-Lite is the better fit for a voice assistant: lower latency, lower cost, quality positioned on par with 2.5 Flash, and it unifies the conversational path with the transaction-extraction calls that already use it, so the whole application now runs one model. This restores the Entry 11 choice for the reasons Entry 11 originally gave.

**This obviates the router from Entry 18 entirely rather than merely deferring it.** Entry 18 deferred a two-model router because, on the free tier, grounding and high request volume could not coexist in one model. On the paid tier they do, since 3.1 Flash-Lite both grounds and has no meaningful daily cap. The router solved a problem that enabling billing removes, so it is dropped from the plan, not postponed, and get_llm_response stays a single-model function with no Gemini-specific routing logic attached.

**Recorded the pricing so the cost tradeoff is explicit, and confirmed it is negligible at this scale.** Paid-tier rates per million tokens are twenty-five cents in and a dollar fifty out for 3.1 Flash-Lite, against thirty cents in and two dollars fifty out for 2.5 Flash, with Google Search grounding free up to five thousand queries a month on the 3.x family and fourteen dollars per thousand after that. A typical Vaani turn is on the order of a thousand input and a hundred output tokens, which puts a thousand rupees of credit at roughly thirty thousand conversational turns, and realistic personal grounding stays inside the free monthly allowance. So cost did not drive the model choice, latency and quality did, with the understanding that token economics would only begin to matter if Vaani served many users.

**Enabling billing introduces a paid-tier dependency that matters the moment Vaani is distributed, now documented for that case.** Mk1 runs on one developer key, so this is invisible today, but anyone else running Vaani needs a billing-enabled Gemini key for grounding to work, because on the free tier the 3.x family cannot ground and 2.5 Flash grounds only about twenty times a day. The README now states this and gives free-tier fallbacks: set the chat model to 2.5 Flash and accept the daily cap, remove the search tool for an uncapped but knowledge-only experience, or route grounding through a separate search API with a more generous free tier such as Brave Search at two thousand queries a month or Tavily at one thousand, paired with a free LLM. That last option is the same provider-agnostic grounding path the Mk2 integration work would naturally build.

## Entry 20: Replaced the Tkinter desktop interface with a mobile-first web UI

**Made a local web UI Vaani's primary front end, superseding Entry 15's desktop interface.** The Tkinter window from Entry 15 was a functional developer-style interface, but the product wants something modern and mobile-first, and Tkinter is a desktop toolkit that cannot be made to look like a phone app. So the front end was rebuilt as a local web UI: a small Python standard-library http.server serves a responsive page and streams pipeline events to the browser over Server-Sent Events, while the existing ui_queue event contract, the same state and transcript messages the Tkinter UI consumed, is reused unchanged. The whole STT, translate, Gemini, translate, TTS pipeline in main.py is untouched; only what listens to the queue changed. The browser starts a session with a plain POST, and the TWS earbud gesture still starts one through the same entry point as before.

**Chose the standard library and Server-Sent Events specifically to avoid a new dependency.** The pipeline only ever pushes events one direction, server to browser, which is exactly what Server-Sent Events do over plain HTTP, and the one thing the browser sends back, a request to start listening, is a single POST. Flask and FastAPI were rejected because they would add a dependency that the working agreement gates, for no capability that the standard library does not already cover here. A real native mobile app was rejected as wrong scope for Mk1, since the backend is Python running on a laptop, so a native app would mean a large networking and packaging effort for no Mk1 benefit. Merely restyling the Tkinter window was rejected because no amount of styling makes a desktop toolkit mobile-first.

**The design is two views behind a deliberately discreet toggle, built around transparency.** The everyday view is a calm chat: the user's words and Vaani's reply, with the English translation shown beneath any non-English turn. The second view is a step-by-step pipeline view that shows exactly what is happening at each stage, split into a live "currently" indicator and a terminal-style transcript that mirrors the prints from a real main.py run. The switch between them is hidden behind a small menu rather than placed in the open, because the pipeline view is for demonstrating and understanding the machine, not for everyday use. The objective driving the whole design is transparency: anyone watching a demo, or the user themselves, should be able to watch the voice travel through each stage rather than treat Vaani as a black box. Visual identity is light and warm with a status orb that changes colour with state, chosen over a dark aesthetic because an assistant should feel approachable. The Tkinter ui.py is kept runnable but is no longer the primary interface.

## Entry 21: Added two intermediate pipeline states so the live view reflects reality

**Emitted two new states and reordered the Speaking signal so the pipeline view shows the true order of work.** The event contract previously emitted only Listening, Thinking, Speaking, and Idle, which is enough for a status label but not for a step-by-step view of the five-stage pipeline. run_conversation now emits a "Translating to English" state before the inbound translation, and vaani_output emits a "Translating back" state before the outbound translation. vaani_output previously signalled Speaking before it did any work at all, including the outbound translation, which would have shown Speaking while the system was still translating, so the Speaking signal was moved to after the translation. Now the order the user sees matches the order things actually happen: listening, translating in, thinking, translating back, speaking.

**These additions are backward compatible by design.** The new state values are additive, and the old Tkinter consumer, which colours known states and falls back to plain grey text for anything else, shows the new labels without error. So the same event stream drives both the new web UI and the old desktop one, and nothing had to change in lockstep. English turns skip both translation states entirely, since there is nothing to translate, and the web view handles the skipped steps gracefully.

## Entry 22: Showed the English translation under non-English turns, not a transliteration

**Chose to render the already-free English translation beneath a non-English turn rather than pay for a transliteration.** In the everyday view a non-English turn shows the native script with a secondary line beneath it, and the question was what that line should be. Two options were weighed: the English translation, or the user's own words transliterated into Latin script. The English translation costs nothing, because the pipeline already produces it as the text it sends to Gemini, so showing it adds no API call. A transliteration would require a new Sarvam call on every non-English turn, paying latency and cost each time, for a line that mainly helps someone who speaks the language but cannot read its script. The translation also reads better for a demo audience who may not read the script at all. Showing both lines was rejected as visually heavy in a chat bubble while still incurring the per-turn transliteration cost. The secondary line appears only on non-English turns, since an English turn has nothing to translate and repeating it would be noise.

## Entry 23: Replaced the transaction-trigger keyword list with an LLM intent classifier

**Switched transaction-intent detection from substring keyword matching to a one-shot LLM classifier, after the logs showed the keyword approach silently failing.** Reviewing two demo logs revealed that in neither conversation was a transaction actually recorded, despite the user asking. The trigger used exact substring matching against a fixed phrase list, and natural phrasings defeated it: "log an expense" does not contain the listed "log expense", and "login expense", an STT garble of the same request, matched nothing. The fix replaces the phrase list with a one-shot classifier on the cheap flash-lite model, run on the English text after inbound translation and kept separate from the chat session so it never enters Vaani's conversational memory. It returns only whether the turn is a request to record a transaction, which is robust to phrasing in a way a fixed list cannot be. Tested against the exact phrases that failed in the logs plus controls that must not trigger, it classified nine of ten correctly.

**The worse half of the bug was the hallucinated success, which the keyword miss exposed.** When the trigger failed, the turn fell through to ordinary chat, and Gemini replied "I have logged that expense" while nothing reached the finance API, so the assistant claimed an action it never took. The classifier fixes the missed trigger, and a separate prompt change in Entry 24 fixes the false claim, but it is worth recording that a brittle matcher does not just miss, it can route a real request into a path that fabricates a result.

**The accepted tradeoff is one extra flash-lite call on every non-exit turn.** This was chosen over two alternatives. Broadening the keyword list is whack-a-mole that keeps missing novel phrasings and still breaks on a bad translation. Giving the chat model a log-transaction tool through function calling is the most elegant long-term shape, since the model would decide when to invoke the flow with no separate call, but it is a larger rework of how the chat turn and the guided transaction flow interact, and was judged too risky to fold in now. The extra call is cheap on flash-lite and small next to the STT, translation, reasoning, and TTS round-trips the turn already makes.

**The one classifier miss is itself an upstream limitation worth recording.** In the Telugu conversation the user said "log an expense" as romanized English embedded in a Telugu sentence, and Sarvam mistranslated it to "log in to spend", which destroys the intent before any classification runs. The classifier correctly reads that English as not a logging request, because it is not one. This is a translation-quality problem with code-mixed input, not an intent-matching problem, and the classifier cannot recover an intent that the translation has already thrown away. Logged for later as a distinct issue.

## Entry 24: Two system-prompt fixes prompted by the demo logs

**Added a prompt guard so Vaani never claims to have performed an action it cannot.** The hallucinated "I have logged that expense" from Entry 23 is a behaviour of the chat model, so the durable fix lives in the system prompt: Vaani cannot record, log, save, or store anything itself, and must never tell the user it has. This belongs at the prompt level because the chat model is what invents the false confirmation. It does not conflict with the real transaction flow, whose confirmation lines like "Done, I've logged four hundred rupees" are hardcoded strings spoken outside the chat model, untouched by the prompt.

**Stopped Vaani ending nearly every reply with an unnecessary question, by rescoping the instruction that caused it.** Both logs showed the same tic: almost every answer ended with a check-in question, "are you following me so far?", "shall I look up the travel time?", even after a one-line calculation. The cause was the prompt itself. An instruction intended only for long explanations split across several turns, telling Vaani to pause and check in before continuing, was being over-applied to every answer. It was rewritten so that most answers end on a statement, a follow-up question is asked only when genuinely needed or when deliberately breaking one long explanation across turns, and the reinforcing "offer to elaborate" line was softened. This is the Entry 4 lesson again: a system prompt is never finished after one draft, and real usage is what exposes which instruction is firing too broadly.

## Entry 25: Hardened the web UI after a Telugu turn crashed it

**Fixed a web-mode crash caused by inherited stdout encoding, which is why the terminal worked but the web UI did not.** The first live web run worked in English and died the moment a turn was in Telugu. The cause: web_ui.py imports main rather than running it as a script, so main's own redirection of stdout to a UTF-8 log file, which only runs under its __main__ block, never activated, leaving stdout as the default Windows cp1252 console. The first print of a Telugu transcript or a rupee sign then raised UnicodeEncodeError, which killed the session worker thread, so the conversation silently died, while the terminal version kept working because its __main__ does tee to a UTF-8 file. The fix reconfigures stdout and stderr to UTF-8 at the top of web_ui.py, with errors set to replace so a stray glyph can never crash a live session. The bug lived in how output was inherited across the import boundary, not in the pipeline, which is the same class of lesson as the earlier type-contract and 503 misdiagnoses.

**Put the web UI on port 8765, not 8000, and made stale instances self-heal so a leftover server cannot block a restart.** The personal finance tracker API that Vaani posts transactions to occupies port 8000, and both must run together during a demo, so the web UI deliberately takes 8765 to avoid the collision. A second problem surfaced repeatedly during development: a leftover instance holding the port made the next launch fail to bind. Two layers now prevent that. A clean exit always releases the socket through a finally block, and startup runs a pre-flight that finds any process still listening on 8765 and terminates it before binding. So a clean shutdown frees the port and a dirty one is cleared by the next launch, scoped to our own port and never the finance tracker's.

**Added line-buffered UTF-8 session logging and silenced benign disconnect noise.** The web UI now tees its own output to a timestamped UTF-8 log under the logs folder, the same pattern main.py and ui.py use, so web-mode conversations are debuggable after the fact. It is line-buffered specifically so a force-quit cannot lose the buffered tail of the log, which matters because a force-quit is exactly the situation where the log is wanted, and because this is the same observability that made diagnosing the transaction and question-tic bugs possible in the first place. Separately, the server now swallows the connection-aborted stack traces that Windows raises when a browser tab closes or the event stream reconnects, since those are normal client disconnects and printing an alarming traceback for each one would look like a failure mid-demo.

## Entry 26: Injected current date and time into the system prompt to fix time hallucination

**Vaani consistently hallucinated the time, and correcting it in conversation did not hold.** A demo log showed it saying "four thirty four in the morning" at ten AM, then correcting to the right time only after the user explicitly said "check the internet," then reverting to the wrong time on the very next turn. The grounded answer lived in the conversation history, but each new turn re-derived the time independently from the model's training context rather than from the correction earlier in the session. The model has no internal clock, so without an authoritative source in the system prompt it was guessing from stale priors.

**The fix is one line in create_chat_session(): stamp the current date and time from the server clock and prepend it to the system instruction before passing it to the model.** The session time is set once at session creation, not re-fetched per turn, so a long session will be slightly stale by the end. That drift is acceptable for a voice assistant where sessions are short, and is strictly better than hallucinating from training data. The model now receives the correct date and time as a hard fact at session start and does not need to search for it.

## Entry 27: Expanded all-caps abbreviations to spoken letter names before TTS

**Bulbul mispronounces financial abbreviations because it reads them as words, not as sequences of letters.** ICICI and HDFC were the first cases observed. The initial fix inserted spaces between letters (ICICI becomes I C I C I), reasoning that the TTS would treat each character as a separate token. It did not: Bulbul received the spaced form and still garbled the output, because it does not interpret space-separated single letters as individual character sounds.

**The correct fix maps each letter to its English spoken name before the text reaches TTS.** A 26-entry dictionary (H to Aitch, D to Dee, F to Eff, C to See, and so on) converts any all-caps word into a sequence of ordinary English words that Bulbul pronounces correctly, so HDFC becomes "Aitch Dee Eff See" and ICICI becomes "Eye See Eye See Eye." The pattern is a regex on any all-caps word of any length, which means new abbreviations are covered automatically with no list to maintain. The expansion is applied only to the TTS input, not to the text printed to the log or shown in the UI, so the log remains readable with standard abbreviation formatting. The spaces approach was an intermediate step that surfaced that Bulbul's character handling is weaker than its word handling, which is useful to know for any future work on the TTS layer.

## Entry 28: Added per-step latency logging across the pipeline

**Added timing to every stage of the pipeline so each session log captures a full latency profile per turn.** The stages timed are user speaking time (how long record_until_silence ran), STT API, inbound translation on non-English turns, intent classifier, Gemini LLM call (added to the existing token-count line), outbound translation on non-English turns, TTS API, Vaani speaking time (actual audio playback across all segments), and total turn time from the moment the user starts speaking to when Vaani finishes. Each prints with a bracketed label and seconds to two decimal places so the full pipeline can be read at a glance without parsing token counts or guessing what happened between log lines.

**The first real timing log surfaced that the intent classifier takes 2.4 to 2.7 seconds per turn, nearly as long as the main Gemini call.** This is a known cost from Entry 23, where the extra flash-lite call was accepted over broadening the keyword list or switching to function calling. The timing data confirms what that decision costs in wall-clock time. The classifier sits between STT and the main Gemini call, so a typical English turn now has roughly 0.5 seconds of STT, 2.5 seconds of intent check, and 3.5 seconds of Gemini before TTS begins. No change to the tradeoff, but the numbers are now in the log rather than estimated.

## Entry 29: Replaced the separate intent classifier with Gemini function calling to eliminate a 2.5-second per-turn cost

**Removed the is_transaction_request() flash-lite call and routed transaction detection through the main Gemini call using function calling, superseding Entry 23.** Entry 28's timing logs confirmed the intent classifier was adding 2.4 to 2.7 seconds on every single conversational turn, not just transaction turns, because it ran unconditionally before the main Gemini call. That made "thinking" time roughly 6 seconds on a typical English turn: 2.5 seconds of classifier plus 3.5 seconds of Gemini. The fix: add a record_transaction FunctionDeclaration to the tools list in create_chat_session(), alongside the existing GoogleSearch tool. Now the same Gemini call that answers the user's question also decides whether the user is asking to log a transaction, by calling the function instead of generating text. get_llm_response() returns None on a function call and the response text otherwise, and run_conversation() branches on that. One API call does both jobs.

**The tradeoff accepted in Entry 23 no longer holds.** Entry 23 rejected function calling as "a larger rework of how the chat turn and the guided transaction flow interact" and too risky to fold in at the time. With the architecture in its current settled state that concern is lower, and the latency data made the cost of the extra call visible enough to act on. The function has empty parameters because the transaction recording flow collects all the fields itself through a guided conversation, so the function call is only a routing signal, not a data carrier. The description in the FunctionDeclaration is the new classifier: it tells Gemini to call the function only when the user explicitly requests to record a transaction right now, not when they merely mention spending money, which replicates the intent the old prompt was trying to capture.

**The one behavioural difference to watch for is that Gemini's function-call decision is now part of the conversational context.** The old classifier ran on a separate stateless model instance, so its yes-or-no verdict never appeared in the chat history. The new approach uses the existing chat session, which means Gemini can, in theory, learn across turns that the user wants to log things, or be confused by a prior function call into calling it again when the user did not ask. That is unlikely with a clear function description, but it is a difference in how state is managed and is worth observing in real use.

## Entry 30: Routed the captured account into payment_source after finding the field it was being written to did not exist

**Fixed a transaction field that was collected on every record and then silently discarded, and found the cause ran deeper than the voice flow.** The guided recording flow listed an account field as required and asked the user which account they paid from on every transaction, but the value went nowhere. The one-shot extraction never returned it, so it was always asked even when the user had already named the account, and the field, called payment_mode, was not part of the payload sent to the finance tracker, so the answer was thrown away before the POST. The first read was a simple wiring gap. Checking the finance tracker's actual schema changed the diagnosis: the transactions model has no payment_mode field at all. Its only account-style field is payment_source, which is optional. So even adding payment_mode to the payload would not have helped, because the tracker uses Pydantic with its default of ignoring unknown fields, which means the POST would have succeeded while silently dropping payment_mode and storing nothing. The field was being written to a name the receiving contract does not have.

**Consolidated the account onto payment_source, the field the tracker actually stores, and made it required.** payment_mode was removed everywhere, from the required-field list, the spoken follow-up question, the single-field extractor, and the payload, and its job was given to payment_source. The account list it carried, ICICI, Jupiter, Slice, Amazon Pay, Mobiquick, and Splitwise Debt, which matches the tracker's real accounts, now drives payment_source extraction, so the account is pulled from the spoken description when the user mentions it and asked as a follow-up only when it is genuinely absent. Vaani had two overlapping account concepts, the constrained payment_mode list and a free-text payment_source whose examples described things like SBI UPI, and the schema settled which one survives: the API has exactly one slot, so they were merged into it.

**Chose required over optional deliberately, against the grain of the schema marking it optional.** The tracker allows payment_source to be null, so the lighter option was to capture the account only when volunteered and never ask for it, removing a question from every transaction. That was rejected because knowing which account a payment came from is much of the point of tracking it, and a ledger where that field is usually empty is far less useful to query later. Because extraction now fills it from the description when the account is named, requiring it does not always cost an extra turn; it only asks when the user genuinely left the account out. A full revert to the Entry 12 four-field flow was rejected for the same reason, since it would have permanently dropped an account the product clearly wanted to capture.

**The general lesson is the one from Entry 2 in a new place: the API boundary is a contract, and a field is only as real as the receiver's schema.** A value can be collected, validated, and sent and still vanish, not because the sending code is wrong but because the name it uses means nothing to the other side, and a permissive receiver that ignores unknown fields hides the mistake rather than surfacing it. The fix was not to send harder but to send to the field that exists. A stale comment near the top of main.py was corrected in passing as well, since it still described the removed is_transaction_request() classifier rather than the function-calling path from Entry 29.

## Entry 31: Fixed transaction function calling, which was silently broken since Entry 29

**Discovered that the record_transaction function call introduced in Entry 29 never fired in production, and traced the failure to two independent causes.** Reviewing a July 2026 session log showed the user asking to record a 3200 rupee porter expense in both Hindi and English across multiple turns, with Vaani responding each time in text: "I have passed that information along" or "the system will record it." No function call log line appeared, and nothing reached the finance tracker. The assistant was hallucinating a handoff it never made, which is exactly the class of failure Entry 23 identified in the keyword-matching design. The fix required two changes, not one.

**The first cause was a direct contradiction in the system prompt.** Entry 24 added a guard that said "you cannot record, log, save, or store transactions yourself; a separate part of the system handles that." This instruction exists to stop Vaani fabricating confirmations, and it does that job, but Gemini also reads it as a reason not to call the record_transaction function, because the prompt says the action is handled elsewhere. The function declaration and the prompt instruction were pulling in opposite directions, and the prompt won. The fix rewrites that line as a positive instruction: when the user explicitly wants to record a transaction right now, call the function silently, without confirming the action in words. The guard against fabricated confirmations stays intact because the instruction still forbids speaking a confirmation, but Gemini is now told to call the tool rather than defer to a fictional external handler.

**The second cause was that passing a FunctionDeclaration alongside a GoogleSearch tool silently disables Automatic Function Calling in the genai SDK.** The SDK printed a warning on every turn saying "Tools at indices [1] are not compatible with automatic function calling (AFC). AFC is disabled." This warning appeared in prior logs but had not been investigated. AFC being disabled does not by itself prevent Gemini from generating a function call signal in its response, but in practice Gemini was not generating one, because the system prompt contradiction had already told it not to. Replacing the FunctionDeclaration with an actual Python callable stub clears the incompatibility at the SDK level and is the approach the SDK warning itself recommends: include Python callables in the tool list and do not include function declarations. AutomaticFunctionCallingConfig with disable set to True is also added to the session config so the SDK does not auto-execute the stub before the main loop can catch the signal via response.function_calls, which is the existing detection path from Entry 29 that remains unchanged.

**The separation between the two causes matters for diagnosis.** Fixing the system prompt alone would have been the minimum viable change if AFC being disabled truly had no effect on whether Gemini generated function calls. Fixing only the callable type would have left the prompt telling Gemini not to use the tool. Both were changed because both were wrong, and a partial fix would have been harder to trust in future debugging. Verified in the first session after the change: the function call log line appeared, the guided recording flow ran end to end in Kannada, the confirmation POST succeeded, and the AFC warning did not appear anywhere in the log.

**A follow-on fix addressed how early in a conversation Gemini calls the function.** With the A+C fix in place, Gemini was calling the function correctly but one turn later than intended. When the user said "I need to record a transaction," Gemini responded in text asking for the amount, category, and other details, then called the function on the next turn when the details arrived. This created a redundant prompt: the guided recording flow always opens with "go ahead and describe the transaction," so the user was effectively asked to describe the expense twice. The root cause was the function description, which said "call only when the user explicitly requests to record right now," which Gemini interpreted as requiring concrete details before the call was warranted. Updating both the function docstring and the system prompt line to say "call as soon as intent is expressed, even if no details have been given, do not ask for details yourself" shifts the trigger to the intent turn, so the guided flow handles description collection on its own with no prior Gemini text turn.

## Entry 32: Two listening fixes — a 30-second recording crash and a noise threshold that was too low

**The session log from 28 June showed record_until_silence() getting stuck for 36 seconds before crashing on a Sarvam 400 error.** The Sarvam STT API rejects audio longer than 30 seconds, but the recording loop had no duration cap. The loop only exits after has_spoken becomes True and then SILENCE_CHUNKS consecutive silent chunks accumulate. If background noise keeps resetting the silence counter, or if has_spoken is never set because no audio crosses the threshold, the loop can run indefinitely past the 30-second limit. The fix adds a max_chunks cap computed from the 30-second limit and CHUNK_DURATION, checked after each chunk is appended. The loop breaks unconditionally at 30 seconds regardless of silence state, so the audio sent to Sarvam is always within the API's accepted range.

**The silence threshold was raised from 0.005 to 0.01 because quiet background noise was registering as speech.** SILENCE_THRESHOLD is the mean absolute amplitude above which a chunk is counted as user speech. At 0.005, about half a percent of peak amplitude, ambient noise, breathing, or microphone hiss could set has_spoken to True and repeatedly reset the silence counter, keeping the loop open well past when the user actually finished speaking. Doubling the threshold to 0.01 still catches normal speech at any reasonable distance but filters out the quiet noise floor that was extending recording windows unnecessarily. This was also a contributing cause of the 36-second case, where the loop appears to have been held open by low-level noise before a genuine speech event finally set has_spoken and then real silence closed it.
